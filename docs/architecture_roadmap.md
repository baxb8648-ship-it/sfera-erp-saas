## 📦 Тема 1: БД на 1 ТБ от строительно-монтажной компании

### Что это даёт
1 ТБ реальных данных — это золото для обучения специализированной локальной LLM.
Если данные содержат сметы, акты КС-2/КС-3, договора, переписку, фото объектов —
ты получишь материал для создания AI, которого **нет ни у одного конкурента**.

### Как правильно работать с такой БД

#### Шаг 1 — Разведка и классификация данных
Перед загрузкой надо понять структуру:
```
Что внутри? → Таблицы / файлы / JSON / PDF?
Какой формат? → SQL-дамп / Excel / 1С выгрузка / CSV?
Чувствительность? → ПД физлиц, коммерческая тайна?
```

#### Шаг 2 — Выбор стратегии использования

**Вариант A: RAG (Retrieval-Augmented Generation)** ← ✅ Рекомендуется
Локальная LLM (Ollama) + векторная база данных (ChromaDB / LanceDB).
БД не обучает модель — она используется как **"внешняя память"**.

```
Вопрос менеджера → Embedding → Поиск в ChromaDB → Top-5 релевантных документов
     ↓                                                          ↓
  Ollama qwen2:7b ←←←←←←← Контекст (5 доков) ←←←←←←←←←←←←←←
     ↓
  Ответ с реальными данными из твоей БД
```

**Преимущества RAG:**
- Не нужно переобучать модель (дорого и долго)
- БД обновляется в реальном времени
- Ответы всегда актуальны
- Работает на ноутбуке / ПК без GPU кластера

**Вариант B: Fine-tuning (дообучение)**
Подходит если нужен специфический стиль ответов (например, писать сметы как конкретный сметчик).
Требует: минимум 1000 размеченных примеров (вопрос → правильный ответ).
Стоимость GPU-часов на RunPod/Vast.ai: ~$50-200 для 7B модели.

**Вариант C: Hybrid** ← для будущего
RAG + дообученная модель на отраслевой терминологии.

## 🤖 Тема 2: Система агентов (Multi-Agent Architecture)

### Что такое агенты и зачем

Агент — это LLM, которой дали **инструменты** (функции) и она сама решает как и когда их вызывать.

Вместо: *"ИИ отвечает на вопросы"*
Получаем: *"ИИ самостоятельно выполняет сложные задачи"*

### Архитектура агентов для ЛЕОНИКА

```
                    ┌──────────────────────┐
                    │   ОРКЕСТРАТОР         │
                    │   (Master Agent)      │
                    │   qwen2:7b / mixtral  │
                    └──────────┬───────────┘
                               │  распределяет задачи
           ┌───────────────────┼───────────────────┐
           ▼                   ▼                   ▼
┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐
│  АГЕНТ ТЕНДЕРЫ   │ │  АГЕНТ СКЛАД     │ │  АГЕНТ ФИНАНСЫ   │
│                  │ │  "Снабженец"     │ │                  │
│ Инструменты:     │ │  Инструменты:    │ │  Инструменты:    │
│ • search_tender()│ │  • get_stock()   │ │  • get_balance() │
```

### Пример работы оркестратора

Менеджер пишет в Telegram:
> *"Нам пришёл новый тендер на АКЗ резервуаров, 3000 м², срок 30 дней, заказчик Оренбургнефть. Подготовь КП, создай задачу для Иванова и добавь заказчика в CRM если его нет"*

Оркестратор (LLM) разбирает задачу и вызывает:
1. `find_client("Оренбургнефть")` → не найден
2. `add_client(name="Оренбургнефть", source="тендер")` → создан ID=47
3. `create_task(title="КП Оренбургнефть АКЗ 3000м²", assignee="Иванов", deadline="+3д")` → создана #156
4. `generate_kp(client_id=47, service="АКЗ", area=3000, deadline=30)` → PDF готов
5. `send_telegram(user="Иванов", file=kp.pdf, text="Твоя задача #156, КП готово")` → отправлено

**Всё это — одна фраза пользователя.**

### Технологии для агентов
- **LangChain / LlamaIndex** — Python-фреймворки для оркестрации агентов
- **Ollama** — локальный LLM сервер (уже есть)
- **Function Calling** — передача инструментов в модель (поддерживается qwen2:7b)

## 📊 Тема 3: Раздел "Управление Проектами" — нужен ли?

### Что уже есть в ЛЕОНИКА CRM
| Функция | Есть? |
|---|---|
| Задачи (Kanban) | ✅ |
| Объекты (строительные) | ✅ |
| Склад + списание по объекту | ✅ |
| Финансы по объекту | ✅ |
| История изменений (Audit Trail) | ✅ |
| Чат по задаче | ✅ |

### Чего не хватает до "взрослого ERP"

**1. Диаграмма Ганта** — визуализация сроков проекта
```
Проект: АКЗ резервуары ОНПЗ
  │ Подготовка поверхности  ████████░░░░░░ 60%
  │ АКЗ слой 1              ░░░░████░░░░░░ 
  │ АКЗ слой 2              ░░░░░░░░████░░
  │ Сдача                   ░░░░░░░░░░░░██
  └─────────────────────────────────────── 
    01.07  08.07  15.07  22.07  29.07
```

**2. Смета / Бюджет проекта**
- Плановая стоимость vs Фактическая
- Материалы: план/факт
- Человеко-часы: план/факт
- Маржинальность в реальном времени

**3. Этапы сдачи (КС-2, КС-3)**
Привязка к этапам акты → автосоздание финансовой транзакции при подписании

**4. Ресурсное планирование**
- Кто из сотрудников занят в какие даты
- Конфликты ресурсов ("Иванов уже на 3 объектах в июле")

**5. Фотоотчёты хода работ**
- Галерея по объекту с привязкой к этапу
- До/После

---

### Рекомендация: Эволюционный путь

> [!TIP]
> **Не нужно делать всё сразу.** Взрослые ERP (1С, Primavera, MS Project) создавались 20 лет. Используй **MVP-подход** — добавляй функции по реальной боли пользователей.

#### Приоритизация по боли:

| Боль | Решение | Сложность |
|---|---|---|
| "Не знаю маржу объекта в реальном времени" | Смета план/факт | 🟡 Средняя |
| "Не вижу кто и где занят" | Ганта + ресурсы | 🔴 Высокая |
| "Некуда прикрепить фото с объекта" | Фотогалерея | 🟢 Низкая |
| "КС-2 подписали — нужна транзакция" | Этапы → Финансы | 🟡 Средняя |
