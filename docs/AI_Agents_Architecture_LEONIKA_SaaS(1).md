## SaaS-платформа ЛЕОНИКА CRM/ERP — мультиагентная система

Версия 2.0 · 30.06.2026 · обновлено по факту аудита `session_context.json`

## 0. Назначение документа

Рабочая спецификация перехода от текущей CRM ЛЕОНИКА к мультитенантной SaaS-платформе с отраслевыми модулями, в каждом из которых работает набор AI-агентов и суб-агентов. Версия 2.0 учитывает реальный аудит проделанной работы и добавляет: (а) принцип «Управление проектами — это слой Core, а не функция одного модуля», (б) девятый модуль — **Мебельное производство**.

### 0.1 Что реально уже работает (аудит на 30.06.2026)

Это важно зафиксировать, чтобы не проектировать заново то, что уже построено.

**Инфраструктура.** Бэкенд FastAPI на локальном ПК, проброшенный SSH-туннелем (Paramiko, автореконнект) на VPS Timeweb Cloud (RU). На VPS — Nginx + Let's Encrypt, домен `api.леоника56.рф`. Фронтенд — статика на Hostiman по FTP, поддомен `срм.леоника56.рф`. Cloudflare используется только как DNS (Proxy: Off, HTTP/3 выключен) — без проксирования трафика. **База данных всё ещё SQLite** — миграция на PostgreSQL с RLS не начата, это по-прежнему главная незакрытая зависимость для мультитенантности.

**Готово и стабильно работает:**
- Kanban-ядро: объекты, задачи, склад, оборудование, финансы, клиенты, Audit Trail с диффом изменений.
- Тендерный парсер: zakupki.gov.ru (live RSS) + 11 коммерческих площадок с fallback-генератором на случай блокировок.
- AI-копилот в чате CRM и Telegram (Ollama qwen2:7b), AI-анализ ТЗ и договоров, AI-фильтр релевантности тендеров.
- ОКВЭД-парсер базы лидов: `okvad_parser.py` + `leads_route.py` — api-fns.ru → rusprofile (обогащение) → Ollama (AI-скоринг), лимит 100 запросов/год на бесплатном тарифе.
- Снайпер-парсер (`subcontract_parser.py`): веб-поиск контрагентов через DuckDuckGo, генерация офферов через Ollama, авто-отправка email с созданием задачи в CRM.
- **Фаза 1 AI Roadmap полностью реализована**: голосовое сообщение в Telegram → Groq Whisper API (`whisper-large-v3-turbo`, облачный, не локальный faster-whisper) → Ollama qwen2:7b извлекает сущности (клиент, контакт, телефон, услуга, объём, дедлайн) → карточка подтверждения с кнопками → авто-создание Client + Task.
- Telegram Mini App с glassmorphism и Haptic Feedback.

**В ожидании:** боевой тест голосового пайплайна и ОКВЭД-парсера, расширение лимита api-fns.ru, Фаза 2 AI Roadmap (авто-ответы на email, авто-генерация КП).

**Важное архитектурное наблюдение:** распознавание речи сейчас идёт через облачный Groq API, а не локально — это отличается от изначального плана «полностью локального голосового конвейера» (Whisper + Ollama + Piper) из исследовательского отчёта. Для MVP это нормально и быстрее в реализации, но при росте тенантов и требований к приватности данных (особенно для модулей Юрист и Холдинг) стоит вернуться к вопросу локального STT.

## 1. Принципы мультиагентной архитектуры

### 1.1 Оркестратор + специализированные агенты

Супервизор-оркестратор (LangGraph) решает, какой модуль/агент обрабатывает событие; отраслевые агенты — узкоспециализированы: один агент = одна ответственность = один понятный результат.

### 1.2 Золотое правило: ручное подтверждение на каждом переходе

Архитектурная константа: переход состояния всегда проходит через узел `human_gate`, который блокирует граф до явного клика человека. Агент готовит результат — не фиксирует действие с последствиями без подтверждения.

### 1.3 Мультитенантность на уровне агентов

`tenant_id` пробрасывается в каждый агент через сессию; у каждого тенанта своя коллекция в векторной БД на каждый модуль.

### 1.4 Управление проектами — сквозной слой Core, а не функция одного модуля

Это отдельно отвечает на вопрос «в каком блоке должно быть управление проектами»: оно не должно быть закопано внутри модуля «Строительство». Технически в текущей CRM Kanban-движок (объекты + задачи + дедлайны + Audit Trail) уже спроектирован как общий, не специфичный для АКЗ/пескоструя — просто пока единственный его потребитель это Строительство.

**Правильная архитектура:** в Core выносится универсальный **Project/Object Engine** — сущность «Проект» с произвольным типом (`object_type`: стройка, мебельный заказ, агро-кампания, договор аренды, M&A-сделка, инициатива холдинга), статусной воронкой, задачами, дедлайнами и привязкой к Audit Trail. Поверх него уже строится модуль-специфичная логика:

- **Строительство** — проект = объект, со сметами КС-2/КС-3, геометками, расходом материалов.
- **Мебельное производство** — проект = заказ на изготовление, с картой раскроя и BOM.
- **Агро** — проект = посевная/уборочная кампания на конкретном поле.
- **Аренда спецтехники** — проект = договор аренды единицы техники.
- **Холдинг** — проект = кросс-подразделенческая инициатива.

Поверх Project Engine в Core имеет смысл единый **PM Copilot Agent** — кросс-модульный агент, который не относится ни к одному вертикальному модулю, а агрегирует статус по всем активным «проектам» независимо от типа: единый дайджест дедлайнов (логика уже частично есть — `scheduler.py` уже шлёт дайджест по тендерам, её нужно обобщить на все типы проектов), эскалация просроченных этапов, сводка «что горит сегодня» для руководителя. Это даёт runtime-выгоду: не нужно писать отдельный «дедлайн-трекер» в каждом новом модуле — Мебель, Агро и Аренда получают его бесплатно, как только подключаются к Project Engine.

## 2. Протокол передачи задач между агентами (Handoff Protocol)

```json
{
  "task_id": "uuid-v4",
  "tenant_id": "leonika-56",
  "pipeline": "tender_to_construction",
  "source_agent": "sales.proposal_agent",
  "target_agent": "legal.contract_review_agent",
  "payload": { "tender_id": 4821, "proposal_doc_url": "/files/kp_4821.docx", "amount": 3200000 },
  "ai_summary": "КП на пескоструй 1000м² для Газпрома, готово к юр.проверке",
  "confidence_score": 0.87,
  "status": "awaiting_confirmation",
  "requires_human": true,
  "confirmed_by": null,
  "created_at": "2026-06-30T10:14:00Z",
  "confirmed_at": null
}
```

Статусная модель: `draft → awaiting_confirmation → approved / rejected → in_progress → completed / escalated`. `escalated` — когда `confidence_score` ниже порога (например < 0.6), агент явно поясняет, что его смутило. Каждый переход логируется в `agent_audit_log` — расширение уже существующей таблицы Audit Trail на агентные действия.

## 3. База знаний (RAG): что нужно собрать для каждого модуля

### 3.1 Общая (Core) база знаний — нужна всем агентам

| Источник | Зачем |
|---|---|
| Карточка компании-тенанта (профиль, ОКВЭД, реквизиты) | Самоидентификация агента |

| Шаблоны исходящих документов | Формат вывода |
| История переписки и сделок | Стиль общения, паттерны решений |

### 3.2 По модулям

| Модуль | Источники для векторизации | Частота обновления |
|---|---|---|
| **Строительство** | Прошлые сметы КС-2/КС-3, ГОСТы/СНиПы по АКЗ/пескоструй/огнезащита, номенклатура с расценками, фотоотчёты | При новой смете / еженедельно |
| **Мебельное производство** | Стандартные форматы листов ЛДСП/МДФ, каталог фурнитуры с артикулами, BOM серийных моделей, история заказов фасадов и сроков поставщиков, регламенты кромкования | При обновлении каталога / при новом заказе |
| **Агро** | Кадастр полей, нормы расхода ГСМ/семян/удобрений на га, агротехнические регламенты, история урожайности | Сезонно |
| **Салоны красоты** | Прайс услуг, расход материалов на услугу, карточки клиентов, расписание | Ежедневно/ежемесячно |
| **Аренда спецтехники** | Паспорта техники, регламенты ТО, история поломок, типовые условия аренды | При добавлении техники / после ТО |
| **Холдинг** | Орг.структура, внутригрупповые договоры, учётная политика консолидации | При изменении структуры |
| **Юрист** | Шаблоны приемлемых условий, реестр судебных дел, due diligence чек-листы, 54-ФЗ и проф. нормативка | Ежемесячно |
| **Продажник** | Библиотека реальных диалогов продаж (датасет и для RAG, и для LoRA), скрипты возражений | Постоянно |
| **Парсер** | Профиль идеального клиента (ICP), чёрный список, конкуренты | По обратной связи продаж |

### 4.2 🪚 Модуль «Мебельное производство» (новый)

Два производственных потока на одной базе склада/фурнитуры/оборудования: **корпусная мебель на заказ из ЛДСП** (индивидуальный раскрой под клиента) и **серийная мебель на металлокаркасе** (производство партиями по BOM).

| Агент | Функция | Вход | Function calling | HITL-точка |
|---|---|---|---|---|
| **Raskroy Agent** (калькулятор раскроя) | Извлекает список деталей (размеры, материал, кромка) из заказа, рассчитывает оптимальную карту раскроя листа ЛДСП минимизируя отход | Спецификация заказа | `extractPanelList()`, `calculateNesting()`, `generateCuttingMap()` | Технолог подтверждает карту раскроя перед отправкой в распил |
| **Cutting Order Agent** | Формирует заказ на распил — своему ЦПУ-станку или подрядчику — отслеживает готовность | Карта раскроя | `createCuttingOrder()`, `trackCuttingStatus()` | Отправка заказа подрядчику |
| **Facade Order Agent** | Заказ фасадов МДФ у стороннего производства, контроль сроков и сверка размеров | Спецификация фасадов | `createFacadeOrder()`, `checkLeadTime()` | Подтверждение заказа у поставщика |
| **Fittings & Stock Agent** | Остатки фурнитуры (петли, направляющие, ручки) и листовых материалов, авто-заявка при минимуме, учёт обрезков для повторного использования | Остатки склада, факт расхода | `checkStock()`, `flagReorder()`, `trackOffcuts()` | Заказ у поставщика подтверждает закупщик |
| **Equipment & Consumables Agent** | Состояние оборудования (пила, кромкооблицовочный станок, ЦПУ), расходники, план ТО | Наработка станков | `predictMaintenance()`, `checkConsumables()` | Остановка станка на ТО — вручную |
| **Production Batch Agent** (для серийной линейки) | Планирует партии по BOM модели, учитывает склад металлокаркасов и ЛДСП, формирует производственное задание | Заказ/прогноз спроса | `calcBOM()`, `scheduleBatch()` | Запуск партии — подтверждение начальника производства |

> Важная деталь по Raskroy Agent: ядро этого агента — это **не LLM-рассуждение**, а детерминированный алгоритм оптимальной упаковки прямоугольников (2D bin-packing / guillotine cutting), как в любом профессиональном раскройном ПО. Роль AI здесь — на входе (распознать спецификацию из текста/файла клиента) и на выходе (объяснить результат, предупредить о высоком проценте отхода). Сам расчёт карты раскроя — классическая алгоритмика, не вызов Ollama.

Этот модуль наиболее зрелый — почти все суб-агенты уже есть как отдельные скрипты. Задача SaaS-этапа — обернуть существующую логику в формальные LangGraph-ноды с единым handoff-протоколом.

## 5. Сквозные пайплайны (примеры межмодульной работы)

### 5.1 Тендер → Продажа → Юрист → Стройка

```
Парсер находит тендер → [Discovery + Enrichment + Scoring]
   ▶ «Взять в работу»
Продажник: Proposal Agent готовит КП
   ▶ «Отправить КП»
Юрист: Contract Review Agent анализирует ответный договор
   ▶ «Договор согласован»
Стройка: Estimate Agent создаёт объект и смету
```

### 5.2 Мебель: заказ → раскрой → производство → склад

```
Клиент оставляет заказ на кухню (через CRM/Telegram, возможно голосом — Voice-to-Lead Agent уже умеет извлекать сущности)
   ▶ Менеджер подтверждает спецификацию
Raskroy Agent рассчитывает карту раскроя ЛДСП
   ▶ Технолог утверждает карту (контроль % отхода)
Cutting Order Agent отправляет заказ на распил
Facade Order Agent параллельно заказывает фасады МДФ у поставщика
Fittings & Stock Agent резервирует фурнитуру под заказ
   → PM Copilot Agent отслеживает сроки готовности всех трёх веток как единый проект
```

### 5.3 Аренда спецтехники → Юрист (страховой случай)

```
Incident Agent фиксирует поломку техники в аренде
   ▶ Диспетчер подтверждает инцидент
Юрист готовит страховое заявление
   ▶ Юрист подаёт заявление вручную
```

### 5.4 Холдинг: перераспределение ресурсов между подразделениями

```
Аренда спецтехники: простой техники 40%
Строительство: дефицит техники на объекте
   → Resource Redistribution Agent обнаруживает дисбаланс
   ▶ Руководитель холдинга утверждает передачу
   → Intercompany Agent формирует акт аренды между подразделениями
```

## 6. Технологический стек

| Слой | Технология | Статус |
|---|---|---|
| Backend API | FastAPI (async) | Работает |
| База данных | SQLite → **PostgreSQL + RLS (план)** | Миграция не начата |
| Оркестрация агентов | LangGraph | Не внедрено, агенты пока как отдельные скрипты |
| LLM (текст) | Ollama / Qwen2:7b | Работает локально |
| STT (голос) | Groq Whisper API (облако) | Работает, план — рассмотреть локальный faster-whisper для приватности |
| Векторная БД | ChromaDB/Qdrant | Не внедрено — следующий шаг для RAG |
| Интерфейс | Telegram Bot + Mini App, веб-CRM | Работает |
| Туннель/инфра | SSH-туннель (Paramiko) на VPS Timeweb + Hostiman FTP | Работает стабильно |
| Аудит | `agent_audit_log` (расширение Audit Trail) | Частично есть, нужно расширить на агентные действия |

## 7. Дорожная карта внедрения

**Этап 1 — фундамент (4–6 недель).** Миграция SQLite → PostgreSQL + RLS (приоритет №1, без неё мультитенантность невозможна). Обобщение Project Engine из текущего Kanban-движка объектов (раздел 1.4) — без этого шага новые модули (Мебель, Агро, Аренда) придётся каждый раз городить свой трекер задач заново. Перевод существующих скриптов-агентов на LangGraph-ноды с единым handoff-протоколом.

**Этап 2 — зрелые модули (2–4 недели, ускорено).** Парсер и Продажник почти готовы — включая Voice-to-Lead, уже работающий в проде. Здесь же — сбор библиотеки диалогов продаж для RAG.

**Этап 3 — Строительство (6–10 недель).** Estimate Agent на основе уже работающего парсинга ТЗ, затем Procurement и Progress.

**Этап 4 — Мебельное производство (6–8 недель, новый приоритет).** Raskroy Agent (алгоритм раскроя — отдельная инженерная задача, не зависит от LLM-инфраструктуры, можно начинать параллельно с другими этапами), затем Fittings & Stock, Facade Order, Production Batch для серийной линейки.

**Этап 5 — Юрист (4–6 недель).** Contract Review Agent расширяется (база уже есть) до Compliance и Litigation. M&A-агент — в последнюю очередь.

**Этап 6 — новые вертикали: Агро, Салоны, Аренда (8–12 недель).** Параллельно, как пилотные ниши.

**Этап 7 — Холдинг (4 недели).** Запускается последним — осмыслен только при наличии нескольких подразделений/модулей у клиента.

## 8. Структура "скилла" агента

### 8.1 Системный промпт
```
Ты — [роль] компании [tenant.name], которая занимается [tenant.industry].
Твоя задача: [узкая ответственность агента].
Ты ВСЕГДА:
- опираешься только на данные из контекста (RAG), не выдумываешь цифры
- если уверенность ниже 0.6 — помечаешь задачу как escalated с пояснением
- никогда не выполняешь действие сам — готовишь черновик для подтверждения
Формат вывода: [JSON-схема / docx-структура / текст]
```

### 8.2 Few-shot примеры — 2–3 эталонных входа/выхода от лучших сотрудников клиента.

### 8.3 RAG-источники — ссылка на конкретную коллекцию из раздела 3, изолированную по тенанту и модулю.

### 8.4 Function calling schema — строго типизированные инструменты, никакого произвольного SQL.

### 8.5 Когда переходить на LoRA fine-tuning — по умолчанию RAG + промпт-инжиниринг; fine-tuning только при 500+ примерах и узком повторяющемся стиле письма (лучший кандидат — Proposal Agent и Script Coach в Продажнике).

## 9. Чек-лист следующих шагов

1. Мигрировать SQLite → PostgreSQL с `tenant_id` и RLS-политиками — без этого ни один новый модуль нельзя продавать как SaaS.
2. Обобщить текущий Kanban объектов в универсальный Project Engine (раздел 1.4) до того, как начинать модуль «Мебельное производство» — иначе будет дублирование логики трекера задач.
3. Создать таблицу `agent_audit_log` и универсальный UI-компонент подтверждения, переиспользуемый всеми модулями.
4. Обернуть `okvad_parser.py` / `leads_route.py` / Снайпер-парсер / Voice-to-Lead в LangGraph-ноды с единым handoff.
5. Спроектировать алгоритм раскроя (Raskroy Agent) как отдельный технический модуль — это классическая 2D bin-packing задача, может разрабатываться параллельно независимо от остальной agent-инфраструктуры.
6. Поднять векторную БД (рекомендация — Qdrant) рядом с GPU-слоем инференса.
7. Решить вопрос локального vs облачного STT (Groq) с учётом требований приватности для модулей Юрист и Холдинг.
